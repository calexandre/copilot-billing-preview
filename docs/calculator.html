Sizing playbook

§ 04.0 · the order to set budgets

Set budgets in this order, derived from the Copilot AI Credits Budget Authorization Spec. The two key facts that drive the playbook: (a) user budgets cap total usage (including pool consumption), while enterprise and cost-center budgets only gate additional spend; (b) cost-center budgets are not quotas — Scenario 6.3 in the spec demonstrates that the same configured CC budget can be active or inactive depending on how the shared pool was consumed earlier in the period.

  1. Get visibility first

    Run the PRU→credits conversion against last month's usage export. You need: total enterprise consumption, P90/P99 per-user consumption, which users are outliers, and which cost centers are heavy. Without this baseline, every other number is a guess.

  2. Enterprise budget = tolerable monthly overage

    This is the top-level dollar lever. Set to $0 if you want the pool to be a hard ceiling. Otherwise, set it to whatever Finance will sign off on. Everything else operates within this guardrail (unless ignoreEnterpriseBudget is used for a specific cost center).

  3. Cost-center budgets only if you need differential overage caps

    If everyone in the org has the same overage tolerance, skip cost centers entirely — enterprise budget plus user budgets do the work. Use cost-center budgets when, say, Platform should be allowed more additional spend than Customer Engineering. Don't expect them to behave like quotas (Scenario 6.3).

  4. Universal default user budget — generous to start

    Rule-of-thumb: pick a value that covers 2–3× a typical user's observed monthly draw. This isn't your spend control — your enterprise budget is. This is your fair-use control. Setting it too low frustrates real work; too high lets one user drain the pool. The default $200 below sits at ~3× the April P75 user.

  5. Override individual budgets for known power users

    Identify them from the usage export. Set them higher than the default — $500, $1,000, whatever makes sense given consumption history. Mind the 10K-budget limit. NOS is well under (1 + 6 + 1 + ~100 = 108 budgets).

  6. Iterate at the end of month one

    Pull the usage export, see who was blocked and why, see what was left on the table, adjust. This is going to be iterative — usage patterns will shift as people learn the new Auto mode and as features evolve. Don't try to nail this on day one.

Worked example · simplified

For the spec's reference 10-user / $190 pool / $310 enterprise budget setup: if observed P50 user draws $25/month and P90 draws $45/month, a universal default around $60–$75 keeps the long tail covered while preventing a single user from solo-draining the $500 total. For NOS scaled up: pool $49k + enterprise cap $15k = $64k total. Divided across 700 seats × safety factor 1.3 ≈ $120/user theoretical max, but the P90 user shape recommends ~$200 universal default in practice.

Practical pitfalls to avoid
  • Setting universal default = per-user license value ($19 / $39, or $30 / $70 in promo) defeats the entire pooling benefit. That's what users contribute to the pool, not what they should be capped at.
  • Setting cost-center budgets very low to act like ring-fenced quotas doesn't work — see Scenario 6.3 in the spec. Quotas don't exist in this model; only overage caps do.
  • Forgetting that code completions and Next Edit Suggestions are free — they don't consume credits and don't count toward any budget. Don't size around them.
  • Forgetting that code review now consumes Actions minutes separately — this is a different budget surface that you may need to revisit, especially at scale.

Per-user blocking · 3-gate check

§ 04.4 · individual test

Each user must pass three gates simultaneously. Failing any one means blocked. The crucial asymmetry from the spec: the user-level gate checks total monthly usage (pool + additional combined), while cost-center and enterprise gates only check the additional-spend portion. So a user can be blocked at the ULB gate even with significant pool remaining — and can be blocked at the CC gate even with significant personal ULB remaining (Scenario 6.3).
